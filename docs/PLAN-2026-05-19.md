## 1. Where we are vs. KittyLitter

### Headcount

| Layer | swe-kitty today | KittyLitter |
|---|---|---|
| iOS SwiftUI files | **18** under `apps/ios/Sources/` | **~115** under `Views/` + **~65** under `Models/` |
| Android Kotlin files | **18** | **124** |
| Shared Rust modules | 4 (`lib`, `transport`, `session`, `views`) | **40+** under `codex-mobile-client/src/` |
| Watch / Live Activity / CarPlay / Catalyst | none | all four shipped |
| Voice (Whisper + realtime) | none | full stack (transcription + WebRTC) |
| Push notifications | none | APNs + FCM proxied via dedicated service |
| Saved state stores (servers / threads / projects / dirs) | one bearer in Keychain/EncryptedPrefs | five typed stores per platform |

We are a *scaffold* of a phone-first coding broker; KittyLitter is a *commercial product*
of approximately the same domain. The gap is structural, not cosmetic.

### The architectural lesson (do this first, or you re-do everything)

Litter's `CLAUDE.md` enshrines a **Rust-first** rule:

> If logic is about session state, thread state, streaming, hydration, approvals,
> auth/account, discovery merge policy, voice transcript/handoff normalization, or
> status normalization, it belongs in `shared/rust-bridge/codex-mobile-client/`.
> Keep Swift/Kotlin thin.

Our current code violates this — `apps/ios/Sources/SessionStore.swift` and
`apps/android/.../SessionStore.kt` both contain reducer-shaped logic, status string
parsing, and reconnect state. The first piece of every package below is **lift
that logic into `core/`** so iOS and Android stop drifting independently.

This isn't a refactor for its own sake — every new feature (voice, push,
multi-server, sessions list) needs a typed shared model the platform views can
project from. Building these features on the current split would create the
exact drift KittyLitter spent two years rolling back.

### 2.3 Voice & audio (the user explicitly named Whisper)

Two distinct rails. Litter ships **both**:

**A. Push-to-talk / one-shot transcription (Whisper-style)**
- `Models/VoiceTranscriptionManager.swift`, Android `VoiceTranscriptionManager.kt`
- `Views/InlineVoiceButton.swift`, Android `InlineVoiceButton.kt`
- `Views/InlineVoiceStatusStrip.swift`, Android `InlineVoiceStatusStrip.kt`
- `Views/AudioWaveformView.swift`
- iOS: Apple `Speech.framework` (`SFSpeechRecognizer`). On-device when supported (iOS 17+ all major locales).
- Android: `android.speech.SpeechRecognizer` with `EXTRA_PREFER_OFFLINE`.
- Output feeds `send_chat`. No protocol change needed.

**B. Realtime voice (full duplex)**
- `Models/RealtimeWebRtcSession.swift`, Android `voice/RealtimeWebRtcSession.kt`
- `Views/RealtimeVoiceScreen.swift`, Android `ui/voice/RealtimeVoiceScreen.kt`
- `Views/VoiceCallView.swift`
- `Views/HomeVoiceOrbButton.swift` (entry point)
- `Models/VoiceRuntimeController.swift`, `Models/VoiceSessionControl.swift`, `Models/VoiceSessionModels.swift`
- `Models/SilentAudioKeeper.swift` — keeps the audio session live in background.
- Uses libwebrtc (stasel/WebRTC SwiftPM on iOS, `io.github.webrtc-sdk:android`). AEC/NS/VAD handled by libwebrtc audio-processing module. Rust owns only signaling + transcript state.

**For swe-kitty:** rail A is high-impact and cheap (no WebRTC dependency, no
protocol change). Rail B requires a broker-side signaling endpoint and a
realtime LLM provider — defer to v1.x unless we wire it through an MCP-style
voice broker that hands the transcribed text to whichever agent is running.

🔴 across the board today.

## 3. Architecture work (do this *before* the feature packages)

### 3.1 Lift the shared session/store model into `core/`

Mirror litter's `shared/rust-bridge/codex-mobile-client/src/store/`. We need:

- `core/src/store/` — `AppStore` analog. Owns snapshots, subscriptions,
  reducer/reconcile logic. Single writer per session, callback-style fan-out
  via UniFFI `callback interface`.
- `core/src/conversation.rs` — typed `ConversationItem` enum: `userMessage`,
  `assistantStream`, `toolCall { kind, status, args, output, diff? }`,
  `subagent { parent, child }`, `handoff { from, to, body }`, `systemNote`,
  `pendingInput { options, kind }`.
- `core/src/hydration.rs` — replay scrollback + memory into typed items on join.
- `core/src/saved/` — typed UniFFI records for servers, threads, projects,
  recent directories; persisted on each platform via Keychain /
  EncryptedSharedPreferences but *shaped* by Rust.
- `core/src/preferences.rs` — adaptive palette, font, density. UniFFI-safe
  Rust records.
- `core/src/discovery.rs` — already a stub; promote to a real merge/dedupe
  policy that consumes platform mDNS seeds.

### 3.2 Tighten the protocol where the new UI needs it

Two extensions to `docs/WEBSOCKET-PROTOCOL.md` (will trigger a frozen-contract
PR — keep it small):

- `view_event { kind: "pending_input", options: […], session_kind: "approval" | "free_text" | "menu" }` — server emits when it detects a prompt; client renders a typed sheet.
- `view_event { kind: "tool_call", phase: "start" | "stdout" | "stderr" | "done", id, name, args?, exit?, duration_ms? }` — replaces the current ad-hoc tool-call string. Backed by KittyLitter parity discussion in `MOBILE-FEATURE-BACKLOG.md` "In Progress".

Both are forward-compatible: clients that don't know the new `kind` ignore it
per §3.3 of the protocol doc.

### 3.3 Tighten the broker for push + background

- `broker/internal/push/registry.go` (new) — store APNs/FCM tokens per pairing.
- `broker/internal/session/watchdog.go` — classify wake-worthy events
  (`stall_alert`, `pending_input`, `exited`, `swap_complete`) and send via the
  push proxy.
- `services/push-proxy/` — small Go service that fan-outs APNs (via
  `sideshow/apns2`) and FCM (via `firebase-admin-go`) given a registered
  token. Litter has its own dedicated service under `services/push-proxy/`
  — we can model the same shape but keep it inside `broker/cmd/swe-kitty-push/`
  to avoid running a separate process.

## 4. Packages

Each package is one merge-train: one or two PRs that ship a coherent slice end-to-end on both platforms.

### Package 1 — **Rust-first refactor** (the foundation)

Scope:
- `core/src/store/` with `AppStore`, snapshots, subscriptions.
- `core/src/conversation.rs` with typed `ConversationItem`.
- `core/src/hydration.rs` replays scrollback + memory into items.
- `core/src/saved/` with `SavedServer`, `SavedThread`, `SavedProject`,
  `RecentDirectory` typed records + platform-persistence callback hooks.
- Both apps switch their `SessionStore`/`AppRoot` to project from the Rust
  store; remove the duplicated state.

Exit criterion:
- iOS and Android render the same typed-item timeline against the same WS
  fixture.
- No `String`-typed status fields on the Swift/Kotlin side.
- `cargo test --workspace` green; CI all green.

Roughly 1 week of focused work. Blocks every package below.

### Package 2 — **Typed conversation rendering**

Scope (iOS + Android in lockstep):
- `Views/Conversation/ConversationTimelineView` driven by `ConversationItem`s.
- Tool-call card with status / args / output / diff / collapse.
- Subagent card (`switch_agent`-aware) — *our* differentiator vs litter.
- Inline handoff view (mid-`switch_agent` UX).
- Diff rendering with per-file / per-hunk grouping.
- Code block view with syntax highlighting (`HighlightSwift` on iOS,
  Prism4j on Android — same libraries litter uses).
- Streaming render coordinator + render cache.

Exit criterion:
- An agent running `bash -c 'ls -la && cat foo.txt'` shows three distinct
  cards (bash command, file read, diff) instead of one undifferentiated
  text blob.
- Mid-swap, the handoff card shows `from`/`to`/`reason`/`body` inline in the
  transcript.

### Package 3 — **Pending input + quick replies**

Scope:
- New `view_event { kind: "pending_input" }` emitted by the broker.
- Per-agent detector strategies in `core/src/pending_input/` (Claude Code
  numbered menus, Codex approval prompts, generic `(y/n)`).
- `PendingUserInputCard` on both platforms — typed buttons, inline in the
  transcript, dismisses when the agent moves on.
- `QuickReplySheet` upgraded from static chips to a sheet that reflects the
  current pending state, with control-key fallback when nothing's detected.

Exit criterion:
- Claude Code asking "1. Yes / 2. Yes, don't ask again / 3. No" renders as
  three native buttons. Tapping inserts the number into PTY.

### Package 4 — **Saved servers + discovery + multi-server UX**

Scope:
- `core/src/saved/server.rs` typed model + platform persistence.
- `Views/DiscoveryView` on iOS, `ui/discovery/DiscoveryScreen` on Android —
  mDNS browser listing every `_swe-kitty._tcp.local` advertiser, plus saved
  servers grouped together with status pills.
- `ServerPillRow` + `ServerPill` reusable components.
- `Views/AlleycatAddServerSheet`-shape add-server flow.
- `Models/NetworkReachabilityObserver` on iOS, `state/NetworkReachabilityObserver` on Android.
- Active-server switcher in the sidebar / drawer.

Exit criterion:
- Pair to two brokeres (laptop + VPS), switch between them with one tap, no
  re-entry of bearer required, status pill shows live health.

### Package 5 — **Push notifications + background reconnect**

Scope:
- `services/swe-kitty-push/` (or inline in `broker/cmd/swe-kitty-push/`)
  with APNs2 + FCM clients, a tiny token registry, and a `POST /push/notify`
  endpoint.
- Broker `POST /push/register` + per-event classifier in `watchdog.go`.
- iOS `PushProxyClient.swift`, Android `LitterFirebaseMessagingService.kt`
  analogs.
- iOS Background App Refresh + Android `WorkManager` periodic reconnect so
  the snapshot is fresh by the time the user taps the notification.

Exit criterion:
- Agent hits a `pending_input` while phone is locked → push notification
  fires with action ("Yes, stop, view") → tap launches into the matching
  session's pending-input card.

### Package 6 — **Voice rail A (Whisper-style transcription)**

Scope (the user explicitly named Whisper):
- `core/src/voice/transcription.rs` — typed transcript stream + handoff to
  `send_chat`.
- iOS `Models/VoiceTranscriptionManager` using `SFSpeechRecognizer` (on-device
  preferred), `Views/InlineVoiceButton`, `InlineVoiceStatusStrip`, `AudioWaveformView`.
- Android `state/VoiceTranscriptionManager` using `SpeechRecognizer` with
  `EXTRA_PREFER_OFFLINE`, same three views.
- Permissions copy in `Info.plist` + `AndroidManifest.xml`.

Exit criterion:
- Tap the mic in the composer, speak, see live transcript chips, release →
  message sent to agent.
- Works offline on iOS 17+ device for `en-US`.

### Package 7 — **Live Activities + lock-screen card**

Scope:
- iOS `TurnLiveActivityController` + `CodexTurnAttributes` → renamed to our
  `SweKittyTurnAttributes`. Shows the active agent + last tool call + elapsed
  on the Dynamic Island / lock screen.
- Android `ui/widget/ActiveTurnWidget.kt` equivalent for AOD.
- `LockScreenCardView` analog.

Exit criterion:
- Run a long agent task, lock the phone, see the activity update on the
  lock screen.

### Package 8 — **iOS visual polish (closes Package B)**

This is the existing Package B sub-plan in `MOBILE-PORT-MATRIX.md`:
- Steps B.2 / B.3 / B.4 / B.5 / B.6 (background, home, settings, in-session
  shell, animated splash). B.1 (Glass primitives) is done.

### Package 9 — **Voice rail B (realtime WebRTC)** — *v1.x*

Defer until after Package 6 ships and proves voice is worth the dependency.
Litter uses libwebrtc; this is a 50MB framework, AEC/NS/VAD complexity, and a
new broker-side signaling endpoint. Pair with an MCP voice broker so any
agent can be voice-driven.

### Package 10 — **Watch / Catalyst / CarPlay** — *v1.x*

Defer. KittyLitter ships these; for swe-kitty the cost/benefit math says
phone + lockscreen Live Activity covers 95% of the value.
